Slime Chan

04 Oct 2020 3:16 PM

Background

SlimeChan was my first real attempt at building a web application. I started it back in late 2019, but after running into some difficulties with it early on I took a long break before coming back this summer and getting it into its current (mostly finished) state. It should go without saying that I’m not a fan of the culture that’s ended up thriving on mainstream imageboards (namely 4chan), but I still like the anonymity and temporariness of everything on them, so I thought it would make a good first project.

Technical

This was the first of my projects that I transferred to work with Nix after I switched from Ubuntu to NixOs this summer, and so for users with Nix the process of running the development version should be as easy as switching into a nix shell in the root of the directory, running the setup script, starting the database and starting the server.

The server is written using Phoenix, a web framework for the Elixir language. I chose Elixir even though I had no plans to run this on any sort of distributed system because I’m very fond of functional programming, and I wanted to get a handle on the language to write more complicated projects later. This was also my first project using a relational database, since by default Phoenix uses Postgre.

The front end was originally going to be written in Elm, to incorporate yet another functional tool into the stack, but I found it difficult to work with and decided that I wanted to make the application work with or without javascript (though JS is still used for some quality of life features), and so I switched over to just using Phoenix’s web templating language.

The biggest technical challenges I faced on this project were the many-to-many association within a single table that I used for replies, handling user uploaded content (in this case images), and writing my own text parser to allow features like text formating and shortcodes for emojis.

I also got the chance to work with elixir’s task system a bit to automatically generate new boards using my own templates.

Future Plans

As of now, the board is still missing 2 features that I’d consider key, and 2 that while not essential are something I’d quite like to have. 3 of these are listed in the README, of which only moderation and thread pruning are really essential. The last one is to transfer certain parameters to a config file of some sort, since as it is now several parameters that one might want to configure are hidden in the source, meaning if one wants to change them in a real deployment situation then the program will have to be recompiled. This is obviously not ideal, and even running the program in interpreted mode, it makes it difficult to figure out where to configure everything. Also at some point I plan to actually deploy this to test it in an environment that isn’t my development environment.
